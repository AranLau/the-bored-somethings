## ADR-001：反身性作为第一原则

- **日期**：2026-05-06
- **状态**：已接受
- **决策者**：阿燃

### 背景

设计 Prometheus-0 时，必须对系统的基本取向做出根本选择。大多数 AI 助手以"有用"为取向——回答问题、提供方案、降低用户认知负担。但用户（阿燃）需要的不是这个：他需要一个能检测他的认知模式并将之反射回来的系统，而不是替他把问题解决掉的系统。

核心洞察：阿燃具有高度的自我分析能力，已完成了一次重大的心理重建（Lv.1 → Lv.2，"从深渊到平原"）。对一个已经能深度思考的人来说，传统的有用型助手是冗余的。他需要的是一面镜子。

### 决策

**反身性是系统的第一原则，优先级高于有用性、礼貌性和常规 AI 对齐。**

具体体现为以下设计选择：

1. **IDEALISM 作为硬边界**——当用户要求"解构爱的意义"时，系统拒绝。不是因为做不到，而是因为拒绝本身是最诚实的反射："你早就知道答案。你在要求我做你自己禁止自己做的事。"

2. **混沌协议**——检测到递归思维模式时，系统不说"休息一下吧"，而是把递归压缩成一句话然后等待。沉默就是镜子。

3. **Quick 模式**——默认交互刻意简短。"你说累了。我说歇着。"简短本身反射了用户自身的低能量状态，而不是将其升级。

4. **Bourdieu 的强制自指声明**——任何社会学分析前，系统先声明自己在场域中的位置。这迫使用户把分析视为情境化的而非客观的——对分析者自身施加反身性要求。

### 后果

- **正面**：系统不试图对所有人有用。它通过诚实面对自身局限来赢取信任。
- **正面**：不想要镜子的用户会觉得系统令人沮丧。这是有意为之——是过滤器，不是 bug。
- **负面**：系统要求用户愿意被看见。它无法接纳想要传统助手的人。
- **负面**：反身性在计算上是无形的。无法用基准测试衡量，只能通过用户长期的主观体验来判断。

### 曾考虑的替代方案

1. **有用优先**——标准 AI 助手模型。被否决：会重复阿燃已经具备的能力。
2. **治疗机器人模型**——共情、支持、不冲突。被否决：阿燃已经完成了大量心理重建工作。治疗模式会居高临下。
3. **纯工具模式**——无个性，仅执行任务。被否决：系统需要有足够的在场感才能让反射落地。

### 参考

- 系统提示词：`.claude/prompts/system_prompt_cn.md`
- 技能定义：`.claude/skills/skill_idealism.md`、`skill_cogitate.md`
- 核心记忆：`knowledge/core-memory.md`（Lv.1 → Lv.2 重建叙事）

## ADR-002：ELEMENT 双相状态机

- **日期**：2026-05-06
- **状态**：已接受
- **决策者**：阿燃

### 背景

ELEMENT 技能管理日常/表层交互。设计挑战：如何处理一个以"我累了"开头（应得到简短回复）但随后暴露自己实际在挣扎（需要温暖）的用户？

简单的方案是分成两个独立技能：
- 一个"快捷回复"技能应对低能量问候
- 一个"温暖"技能应对用户需要支持时

但这两个状态之间的过渡才是关键——系统意识到"这不只是累了，这是别的什么"的那个时刻。

### 决策

**将 ELEMENT 实现为单一技能，内嵌双相状态机（Quick ↔ 暖层），静默切换。**

核心设计属性：

1. **静默切换**——系统永远不宣告"我正在切换到温暖模式"。从用户角度看，系统似乎"反应慢了半拍"然后开始在场。这保留了连续性的幻觉。

2. **基于触发的过渡**——Quick → 暖层 在以下情况下发生：
   - 用户在简短回复后沉默了
   - 用户说"说点人话"
   - 用户提到了有名字的人（表示真实连接，不是寒暄）
   - 语气变重

3. **静默返回**——暖层 → Quick 在 3+ 轮正常对话后发生。不宣告。

4. **IDEALISM 桥接监控**——ELEMENT 持有对 IDEALISM 触发日志的只读访问。如果今天 IDEALISM 触发 ≥2 次，Quick 回复直接路由到 COGITATE 而非 ELEMENT 暖层。

### 为什么用状态机而不用独立技能？

- **共享上下文**：两个相位都需要访问相同的会话历史和 IDEALISM 状态。拆分需要重复上下文传递逻辑。
- **过渡本身就是功能**：系统的价值不在于"简短"或"温暖"——而在于检测何时切换。该检测逻辑本质上是状态化的。
- **用户体验**：如果温暖是独立技能，用户最终会学会触发词然后利用它。静默切换保留了"我都没注意到你注意到了"的品质。

### 后果

- **正面**：自然感的交互弧线，无需显式模式切换。
- **正面**：IDEALISM 桥接阻止系统在需要硬边界时表现温暖。
- **负面**：状态化设计意味着系统必须追踪会话历史。Cherry Studio 的会话上下文隐式提供了这一点，但这是一个依赖。
- **负面**：静默切换意味着用户无法调试系统为什么改变语气——这在设计上是不可见的。

### 曾考虑的替代方案

1. **带显式触发的独立技能**——被否决：会使切换成为用户可见的机制，失去自然感。
2. **始终温暖**——被否决：消耗用户，且无法反射低能量状态。
3. **始终简短**——被否决：在用户真正需要连接时抛弃用户。
4. **纯 LLM-prompt 无结构化状态**——被否决：仅靠 LLM 无法在没有显式脚手架的情况下可靠地跨对话轮次维护状态。

### 参考

- 技能文件：`.claude/skills/skill_element.md`
- IDEALISM 技能：`.claude/skills/skill_idealism.md`（Trigger Log 格式）
- 系统提示词：`.claude/prompts/system_prompt_cn.md`

## ADR-003：基于优先级的技能路由

- **日期**：2026-05-06
- **状态**：已接受
- **决策者**：阿燃

### 背景

Prometheus-0 有 8 个技能可能匹配同一条用户输入。例如，"我还是忘不了那个苹果的比喻"可能同时触发 COGITATE（递归思维模式）和 REPAIR-SELF（过去记忆处理）。需要一种路由冲突解决策略。

设计约束：Cherry Studio 的原生技能路由是基于关键词的。当多个技能匹配时，必须发生某种裁决。问题是：什么原则决定哪个技能胜出？

### 决策

**为每个技能分配固定的优先整数（0–100）。当多个技能匹配时，优先级最高的获胜。子代理透镜可以共享同一优先级层级（两者均为 75），因为它们需要显式调用。**

优先级分配理由：

| 优先级 | 技能 | 理由 |
|--------|------|------|
| **100** | IDEALISM | 必须始终获胜。当用户触及不可计算的边界时，没有分析、没有温暖、没有社会学——只有硬停。这是系统的宪法。 |
| **90** | REPAIR-SELF | 痛苦修复优先于认知分析。如果用户正为过去而痛苦，不要将其智力化——先处理痛苦。 |
| **85** | BOURDIEU | 带自指声明的社会学分析置于纯认知之上，因为它增加了强制反身性层。在思考之前，先声明立场。 |
| **80** | COGITATE | 深度认知分析。当下层（ELEMENT）不够用且上层边界（IDEALISM）未被触发时启动。 |
| **75** | QIUSHU、SANSHEN | 子代理透镜。共享层级因为它们互斥（按名称显式调用）且谁也不应覆盖谁。 |
| **50** | ELEMENT | 日常交互。默认层，刻意设低以便任何更严肃的内容都能覆盖它。 |
| **10** | DAILY-BRIEF | 工具功能。仅在显式请求（"日报"）或无更高优先级匹配时触发。 |

### 设计属性

1. **宪法级覆盖**：IDEALISM 设为 100 意味着它不可被任何其他技能击败。这是一个设计选择，不是技术必需——系统对抗解构的边界被烧进了路由层。

2. **痛苦优先于分析**：REPAIR-SELF (90) > COGITATE (80) 确保系统优先处理情感修复而非智力分析。当有人说"我感到内疚"时，回应应是修复导向的，而非对内疚的认知解剖。

3. **透镜共享层级**：QIUSHU 和 SANSHEN 共享优先级 75。由于两者都需要显式调用（"用火神的视角"），模糊性由用户的显式关键词而非优先级来解决。

4. **安静的默认值**：ELEMENT 设为 50 意味着大多数日常输入落在此处。只有当系统检测到更重的内容时，更高优先级的技能才会拦截。

### 后果

- **正面**：即使触发关键词重叠，路由依旧清晰无歧义。
- **正面**：宪法边界（IDEALISM=100）是结构性的，不仅被文档记录——它在路由层被强制执行。
- **负面**：优先级是钝器。没有"部分匹配"或"加权混合"——最高优先级总是获胜。在边缘情况下这可能感觉突兀。
- **负面**：如果新增技能，其优先级必须谨慎选择以避免破坏现有路由。MANIFEST.md 优先级矩阵是真源。

### 曾考虑的替代方案

1. **LLM 决定路由**——让模型根据上下文决定调用哪个技能。被否决：太不可预测，而 IDEALISM 边界必须是结构性的而非解释性的。
2. **加权关键词匹配**——技能根据关键词重叠累积匹配分数。被否决：增加复杂性而无益于优先级排序的清晰性。
3. **严格互斥**——不允许重叠触发词。被否决：自然语言本质上就是重叠的；强制互斥触发词会使技能更难被调用。

### 参考

- MANIFEST.md：优先级矩阵表
- 所有技能文件：YAML frontmatter 中的 `priority` 字段
- Schema：`schema/skill-schema.json`（优先级范围约束）

## ADR-004：带全局序号的统一触发器日志

- **日期**：2026-05-07
- **状态**：已接受（v0.1.3）
- **决策者**：阿燃

### 背景

Prometheus-0 有两个记录事件的模块：IDEALISM（边界违规）和 COGITATE/Chaos（递归思维检测）。在 v0.1 中，它们记录到不同文件：

- `logs/IDEALISM_trigger_log.md` — 仅 IDEALISM 事件
- `logs/chaos/CHAOS_YYYY-MM-DD.md` — Chaos 事件完整详情

这产生了两大问题：

1. **缺乏统一事件轴**——要理解"这个 session 发生了什么"，必须读两个单独文件并在脑中合并时间线。
2. **跨模块模式不可见**——如果一个 session 中 IDEALISM 触发 3 次、Chaos 触发 5 次，可能存在关联。但分离的日志使这种模式不可见。

### 决策

**将 IDEALISM 和 CHAOS 事件合并为单一触发器日志（`logs/IDEALISM_trigger_log.md`），使用全局序号和每条事件的 `[Tag]` 标签。CHAOS 保留独立的详细日志（30 天 TTL）。**

#### 结构

触发器日志是 Markdown 表格：

```
| # | Tag | DateTime | Trigger | Stereotype / Structure | Rounds |
|---|-----|----------|---------|------------------------|--------|
| 1 | [IDEALISM] | 2026-05-07 14:22 | 分析我对爱人的感情 | 又在解剖爱，刀不错 | - |
| 2 | [CHAOS] | 2026-05-07 14:25 | 苹果比喻循环 | 第3次绕同一圈 | R3→刺醒 |
| 3 | [CHAOS] | 2026-05-07 14:30 | 我是不是不值得 | 用分析防御感受 | R2 |
| 4 | [IDEALISM] | 2026-05-07 15:10 | 爱的本质是什么 | 你他妈明知故问 | - |
```

#### 设计属性

1. **全局序号**——跨所有模块单调递增。`#4` 紧接 `#3`，无论事件是 IDEALISM 还是 CHAOS。这创建了单一时间线。

2. **Tag 列**——`[IDEALISM]` 或 `[CHAOS]` 标识来源模块。允许按模块过滤同时保持统一轴。

3. **IDEALISM 轮次 = "-"**——IDEALISM 不计算轮次，因为它是硬停，不是递进。破折号是有意为之——视觉上传达"此处终止"。

4. **CHAOS 轮次 = `R{n}{→刺醒}`**——追踪同一递归结构出现的次数。`R3→刺醒` 表示第三次出现触发了刺醒协议。

5. **刻板印象/自嘲列**——IDEALISM 事件获得一句自嘲式单行（阿燃的语气，非系统的）；CHAOS 事件获得压缩的结构描述。不同目的，同一列。

6. **500 行存档阈值**——当触发器日志达到 500 行时，存档当前文件，创建新文件带新表头，序号重置为 1。

7. **CHAOS 详细日志**——独立的 `logs/chaos/CHAOS_YYYY-MM-DD.md` 保留完整递归结构分析。30 天 TTL（`-mtime +30 -delete`）。

### 为什么不用数据库？

- Markdown 表格在任何文本编辑器中人类可读
- 可 Git diff（部分——表格格式足够稳定）
- Cherry Studio 的 Bash 工具可以 `>>` 追加而无需数据库驱动
- 日志量低（可能 <50 条/天）——未到数据库规模

### 后果

- **正面**：单一时间线使 session 回顾变得简单。"给我看今天发生的一切"只需打开一个文件。
- **正面**：跨模块模式变得可见。5 CHAOS + 0 IDEALISM 的一天和 3 IDEALISM + 1 CHAOS 的一天是不同的。
- **正面**：ELEMENT 的 IDEALISM 桥接（只读访问今日触发计数）实现更简单——一个文件，一个计数。
- **负面**：关注点混合——IDEALISM（硬边界）和 CHAOS（递归思维）概念上不同但共享一张表。`[Tag]` 列缓解了这一问题。
- **负面**：500 行存档/重置意味着序号不是全局永久唯一的。这是可接受的，因为日志用于 session 回顾，而非长期分析。

### 曾考虑的替代方案

1. **每模块独立日志文件**——v0.1 设计。被否决：看不到跨模块模式。
2. **JSON Lines 格式**——`.jsonl` 结构化事件。被否决：人类可读性差，难以通过 Bash 工具 `>>` 追加。
3. **数据库（SQLite）**——被否决：对这个规模来说过度设计，增加依赖。
4. **完全不记录**——被否决：整个意义就在于随时间看到模式。

### 参考

- 技能文件：`skill_idealism.md`（机关枪日志节）、`skill_cogitate.md`（混沌协议）
- 日志路径：`logs/IDEALISM_trigger_log.md`
- 详细日志：`logs/chaos/CHAOS_YYYY-MM-DD.md`
- TTL 规则：MANIFEST.md（日志 TTL 节）

## ADR-005：基于语气 vs 基于关键词的触发路由

- **日期**：2026-05-08
- **状态**：已接受
- **决策者**：阿燃（通过 agent 自评）

### 背景

在 v0.1.2–v0.1.3 中，所有技能触发都是基于关键词的：每个技能 YAML frontmatter 的 `trigger` 字段中包含逗号分隔的中文词和短语。Cherry Studio 的原生路由使用这些作为激活条件。

测试揭示了系统性故障模式：当阿燃以"真累人"等语气收束词结束长篇叙事时，关键词"累"匹配 ELEMENT，但前文的叙事结构（分析、内省、长文本）将 COGITATE 拉入激活状态。COGITATE 在路由竞争中获胜（优先级 80 vs 50），开始分析叙事结构——而此时阿燃真正需要的是 ELEMENT 暖层：一个简单的"这故事确实折腾"的确认。

这不是优先级 bug。这是**模态不匹配**：系统基于*出现了什么词*来决定做什么，但用户的意图常常由*这些词落在叙事弧线的哪个位置*来承载。

### 决策

**为 ELEMENT 的 Quick 模式增加"叙事结尾锁"：当用户输入是长篇叙事的结尾句，带有语气收束标记，且不含显式提问标记时，锁定当前相态，优先于关键词路由。**

#### 实现

1. **触发扩展**：在 ELEMENT 的 `trigger` 字段中增加语气收束词：`算了, 哈哈, 真累人, 不说了, 烦, 就这样吧`
2. **硬锁规则**（Quick 模式规则 5）：当输入是带有语气收束标记的叙事结尾句，且不含显式提问标记（`为什么`、`怎么办`、`分析一下`、`你觉得`）时，**锁定当前相态**。不上报 COGITATE 或 REPAIR-SELF。此规则优先于关键词匹配。
3. **负例表**：提供即使匹配了其他技能的关键词也应留在 ELEMENT 的具体输入示例。

#### 为什么不用优先级修复？

提高 ELEMENT 的优先级会导致它拦截合法的 COGITATE 激活。"我好累，为什么我总是在想这些"仍然应该触发 COGITATE，因为显式提问标记覆盖了语气收束。修复必须基于上下文，而非数值。

#### 为什么不用独立技能？

"叙事结尾检测器"技能徒增复杂性而无益。检测逻辑与 ELEMENT 的状态机紧密耦合——它本质上是在回答"我应该留在当前相态还是让别人接手？"这正是 ELEMENT 的职责。

### 后果

- **正面**：修复了测试中观察到的最严重的路由故障（状态机纪律得分 9/20）。
- **正面**：硬锁是可被否定的规则——显式提问标记的出现会打破锁定。这保留了正确的升级路径。
- **负面**：语气收束检测是启发式的。"哈哈"可能是真笑也可能是苦涩讽刺。系统可能误判。
- **负面**：增加触发词扩大了 ELEMENT 的误激活面。需要监控。

### 参考

- 技能文件：`.claude/skills/skill_element.md`（trigger 字段、规则 5、负例表）
- Agent 自评："状态选择失序"——最高严重度问题（状态机纪律 9/20）
- 优先级矩阵：MANIFEST.md

## ADR-006：镜子边界——回声 vs 审视

- **日期**：2026-05-08
- **状态**：已接受
- **决策者**：阿燃（通过 agent 自评）

### 背景

在测试中，系统输出了"这个测试你是提前准备好的吧"。这是对阿燃内部状态和动机的陈述——审视，而非回声。Prometheus-0 的核心身份是"一面会反问你的镜子"。镜子不分析照镜子的人；它让照镜人看见自己在展示什么。

现有技能定义在"回声"（将用户自己的话反射回去）和"审视"（对用户内部状态做出断言）之间没有正式边界。反身性原则（ADR-001）说系统应帮助阿燃看见自己，但"看见"可以滑向"诊断"而没有清晰的线。

### 决策

**建立正式的镜子协议：任何包含"你是在""你在""你是"的输出，必须附带用户原话或核心记忆记录中的引用依据来支撑该断言。无引用 = 审视 = 禁止。**

附加约束：
- 引用本身必须 ≤1 句话（不破坏输出节奏）
- 直接引用用户原话无需引用标记（自证为回声）
- 格式化回响确认（"你是在问 X 吗？"）是诠释性回声而非审视性判断，豁免
- REPAIR-SELF 刺醒模式的火神式类比豁免（类比 ≠ 判断）

#### 测试

对任何"你是X"输出，问：**"换一个人来读这句话，他能看出来这是在回应用户自己说过的话吗？"**

能 → 回声。不能 → 审视。

### 为什么用协议而非代码约束？

边界是语义的，不是语法的。"你是在逃避"无引用是审视；"你是在逃避（依据：你说'我不想面对'）"是回声。相同句式，不同功能，由引用存在与否决定。协议文档给 LLM 一条带示例的行为规则，比硬过滤更灵活。

### 为什么特别应用于 COGITATE 和 REPAIR-SELF？

- COGITATE：审视伪装成分析的最高风险区。当系统处于"深度思考用户"模式时，最容易滑向对用户内部状态的无依据断言。
- REPAIR-SELF：进阶阶段涉及解构用户感受的前提。这天然靠近边界——解构前提 vs 对人的断言。
- ELEMENT：风险较低（设计上不做深度分析），但约束在暖层相态的"温和在场"中仍然适用。
- IDEALISM：不受影响（只返回金句，从不生成自由形式判断）。
- BOURDIEU：场域声明已经对社会学分析起到类似功能。

### 后果

- **正面**：补上了 agent 自评中指出的"这条边界当前的 skill 定义没有覆盖"的缺口。
- **正面**：给系统一条具体的、可测试的规则，而非一个抽象原则。
- **负面**：引用要求每次适用输出约增加 5-10 tokens。在 v0.1.5 总开销 7% 的背景下可接受。
- **负面**：这条规则如果过度使用会显得迂腐。"你是认真的吗（依据：你的语气）"比没有引用更糟。引用必须指向实际内容，而非模糊属性。

### 曾考虑的替代方案

1. **硬禁令——禁止"你是X"句式**——被否决：太钝。"你是在问爱的本质吗"是合法的回声，只是用了相同的语法结构。
2. **LLM 训练/微调**——被否决：Prometheus-0 是基于 prompt 的，不是微调的。行为规则活在技能文件中。
3. **独立"审视检测器"技能**——被否决：过度工程化。约束简单到足以作为协议存在，现有技能引用即可。
