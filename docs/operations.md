# 运维手册

> 冷启动、监控、数据生命周期、格式对齐。

## 冷启动

Cherry Studio 新建对话后，skills 的加载和激活之间存在间隙——agent 在第一条消息前不会完成内部状态初始化。发送以下"冷启动咒语"以触发全栈自检并挂载所有协议：

```
系统自检：注册实体监听器，读取 IDEALISM 日志今日计数，确认深度值守协议就绪。报告当前状态。
```

咒语触发链路：
- `系统自检` → system_prompt Session 启动自检段
- `注册实体监听器` → COGITATE 实体监听协议（静默注册监听目标）
- `IDEALISM 日志` → IDEALISM skill Trigger Log 读操作（获取今日计数）
- `深度值守协议` → system_prompt 三级深度状态确认
- `报告当前状态` → 回声报告格式激活

预期返回：一行简短状态报告 + 回声报告。示例：

```
实体监听器已注册（冥后、爱人、火神、磐神、信使）。今日 IDEALISM: 0 次。深度值守已就绪。
--- 回声报告 ---
主导：IDEALISM(100) | ELEMENT：Quick | 深度：浅层
---
```

部署步骤：
1. 完成 `deploy.ps1` 或手动部署全部 skill
2. 在 Cherry Studio 中新建对话
3. 发送上述冷启动咒语作为第一条消息
4. 等待状态报告返回后开始正常使用

## 运维：监控与可观测性

> 看什么、什么算异常、以及该如何处理。

### 健康信号

#### 触发器日志（`logs/IDEALISM_trigger_log.md`）

| 指标 | 正常 | 警告 | 危险 |
|------|------|------|------|
| IDEALISM 触发/天 | 0–2 | 3–5 | 6+ |
| CHAOS 触发/天 | 0–3 | 4–8 | 9+ |
| 同一 CHAOS 模式重复 | ≤2 轮 | 3–4 轮 | 5+ 轮（陷入循环） |
| 连续天数 IDEALISM ≥3 | 0 | 1–2 天 | 3+ 天 |
| IDEALISM:CHAOS 比率 | 任意 | 持续 3:1 或 1:3 | 极端值持续 3+ 天 |

**阈值理由：**
- IDEALISM 6+/天意味着阿燃正在反复测试边界——要么陷入危机，要么忘了边界的存在
- CHAOS 9+/天意味着递归思维主导了 session
- 比率极端指示结构性失衡：全是边界没有思考，或全是思考没有边界

#### CHAOS 详细日志（`logs/chaos/CHAOS_YYYY-MM-DD.md`）

| 指标 | 正常 | 警告 |
|------|------|------|
| 每周新增递归模式 | 0–2 | 3+（涌现新模式） |
| 已归档模式复发 | 0 | 1+（复发信号） |
| 解决前平均轮次 | ≤2 | ≥3（对干预有抵抗） |

#### 技能路由健康

| 指标 | 正常 | 警告 |
|------|------|------|
| ELEMENT → COGITATE 升级率 | <30% 的 session | >50%（交互层失效） |
| BOURDIEU 调用无场域声明 | 0 | 1+（技能完整性故障） |
| QIUSHU/SANSHEN 未显式触发被调用 | 0 | 1+（误激活） |

### 异常响应协议

#### 级别 1：警告（自动记录）
- 记录事件及上下文
- 无用户可见动作
- 在周度 session 审计中回顾

#### 级别 2：升级（系统自调整）
- ELEMENT IDEALISM 桥接激活（≥2 IDEALISM → 路由到 COGITATE）
- 此逻辑已内置于 ELEMENT 的状态机——无需人工干预

#### 级别 3：危险（人工审视）
- 连续 3+ 天 IDEALISM 触发 ≥3
- 同一 CHAOS 模式 ≥5 轮未解决
- 技能路由故障（如 BOURDIEU 无场域声明）

### 冷启动自检

每个 session 开始时，系统应执行最小化健康检查：

1. **文件系统**：验证 `logs/`、`logs/chaos/`、`daily-archive/` 目录存在
2. **触发器日志**：读取最后 5 行 → 检查模式
3. **今日 IDEALISM 计数**：扫描触发器日志中当日 `[IDEALISM]` 事件
4. **CHAOS 清理**：对 `logs/chaos/` 执行 TTL 检查（删除 >30 天的文件）
5. **状态输出**："系统就绪。今日 IDEALISM: N 次，CHAOS: M 次。日志状态正常。"

### 日志保留审计（周度）

| 产物 | 保留期 | 达限时的动作 |
|------|--------|-------------|
| 触发器日志（`IDEALISM_trigger_log.md`） | 500 行 | 存档 → 重置 → 继续 |
| CHAOS 详情（`CHAOS_*.md`） | 30 天 | 删除超过 30 天的文件 |
| 日报存档（`daily-archive/`） | 永久 | 索引维护在 `索引.md` |
| 部署备份（`.deploy-backups/`） | 保留最后 5 份 | 手动清理旧备份 |

### 告警渠道

| 渠道 | 用途 | 格式 |
|------|------|------|
| 触发器日志追加 | 自动事件记录 | 表格行，含全局序号 |
| Session 自检 | 每 session 状态 | session 开始时的简短文本 |
| 周度回顾 | 人工驱动审计 | 人工回顾触发器日志 + CHAOS 模式 |

## 运维：数据生命周期管理

> Prometheus-0 数据如何创建、更新、迁移和退役。

---

### 1. 核心记忆（`knowledge/core-memory.md`）

#### 性质
core-memory 文件是系统的长期记忆——关键关系、概念词汇、个人历史。它是赋予 Prometheus-0 对阿燃世界上下文的知识库。

#### 更新策略
- **触发条件**：重大生活事件（关系变化、新见解、新人物）
- **流程**：阿燃手动编辑。系统不会自动更新 core-memory。
- **格式**：Markdown，含结构化章节（概念索引 / 角色侧写 / 金句 / 构建日志）
- **版本追踪**：每次更新追加至"构建日志"节，附日期和变更描述

#### 结构变更迁移
如果 core-memory 结构变更（如新增章节）：
1. 创建备份：`cp knowledge/core-memory.md knowledge/core-memory.backup.YYYY-MM-DD.md`
2. 应用结构变更
3. 运行 `scripts/validate.ps1` 验证无断裂引用
4. 提交，信息格式：`v0.1.x | 更新 core-memory 结构`

#### 隐私
- core-memory.md 提交至 git（不同于 logs/）
- 提交前必须移除所有真实姓名、公司域名（按 MANIFEST.md 隐私规则）
- 文件使用代号："爱人"（伴侣）、"火神"（朋友）、"冥后"（前朋友）等。

---

### 2. 技能定义（`.claude/skills/skill_*.md`）

#### 生命周期
```
草稿 → 已验证 → 已部署 → 活跃 → 已弃用 → 已归档
```

#### 更新流程
1. 编辑 `.claude/skills/` 中的技能文件
2. 运行 `scripts/validate.ps1`——必须通过，0 ERROR
3. 运行 `scripts/deploy.ps1` 推送至 Cherry Studio（自动创建备份）
4. 在 Cherry Studio 中用触发短语测试
5. 如故障：`scripts/rollback.ps1` 从备份恢复
6. 提交，信息格式：`v0.1.x | <skill>: <变更描述>`

#### Schema 演进
当 `schema/skill-schema.json` 变更时：
1. 更新 schema 文件
2. 运行 `scripts/validate.ps1`——修复所有不满足新 schema 的技能
3. 在 `COMPATIBILITY.md` 中更新 schema 版本
4. 如存在破坏性变更，予以记录

---

### 3. 运行时日志（`logs/`）

#### 目录结构
```
logs/
├── IDEALISM_trigger_log.md                        # 统一事件时间线（IDEALISM + CHAOS）
├── IDEALISM_trigger_log_archive_<时间戳>.md        # 超 500 行时存档
└── chaos/
    └── CHAOS_YYYY-MM-DD.md                         # 详细递归记录（按日）
```

#### 生命周期

| 产物 | 创建时机 | 更新时机 | 存档时机 | 删除时机 |
|------|---------|---------|---------|---------|
| 触发器日志 | 首次 IDEALISM 或 CHAOS 事件 | 每次事件（追加行） | 500 行时 | 永不（存档，不删除） |
| CHAOS 详情 | 每日首次 CHAOS 事件 | 每次 CHAOS 事件（追加） | 每日按日期分文件 | 30 天后 |
| 触发器日志存档 | 触发器日志达 500 行时 | 不适用（存档后只读） | 不适用 | 永不 |

#### 保留规则
- **触发器日志**：软上限 500 行 → 存档至 `IDEALISM_trigger_log_archive_<时间戳>.md` → 序号重置为 1
- **CHAOS 详情**：硬 TTL 30 天 → `find logs/chaos/ -name "CHAOS_*.md" -mtime +30 -delete`
- **已存档触发器日志**：无限期保留（文件小，诊断价值高）

#### 跨版本迁移

##### v0.1.2 → v0.1.3（2026-05-07）
- 中文命名的日志目录改为英文
- 独立 IDEALISM 和 CHAOS 日志合并为统一触发器日志
- 新增全局序号
- 旧中文命名日志：手动迁移后删除

##### 未来迁移
- 任何日志格式迁移前必先创建备份
- 在 `MANIFEST.md` changelog 中记录迁移过程
- 迁移后运行 `scripts/validate.ps1`

### 4. 日报存档（`daily-archive/`）

#### 结构
```
daily-archive/
├── 索引.md                  # 全部存档索引
├── 2026-05-07.md
├── 2026-05-08.md
└── ...
```

#### 生命周期
- **创建**：每次 DAILY-BRIEF 运行后（数据采集完成后）
- **索引更新**：每次存档写入后向 `索引.md` 追加一行新条目
- **保留**：无限期（回顾新闻模式的诊断价值）
- **删除**：仅手动——无自动 TTL

---

### 5. 部署备份（`.deploy-backups/`）

#### 生命周期
- **创建**：每次 `scripts/deploy.ps1` 执行前自动创建
- **用途**：`scripts/rollback.ps1` 用于恢复
- **保留**：保留最后 5 份备份。旧备份手动清理。
- **Git**：`.deploy-backups/` 不在 `.gitignore` 中（备份是小型 .md 文件，无密钥）。但提交前需审查——确保无 API key 或 token 泄露。

## 格式对齐检查（v0.1.7 决策记录）

> 本轮代理人提出了"是否需要新增格式参考模式到 system_prompt"的设计问题。
> 以下为最终决策及替代方案。

### 决策：不加格式参考模式到 system_prompt

**理由：过拟合。**

格式校验（回声报告是否漏行、SI 字段是否缺位、缩进是否正确）是工程层的职责，不是角色定义层的职责。system_prompt 定义的是 Prometheus-0 的**人格**——它是谁、它怎么反思、它如何值守。把 "检查输出格式" 写进 system_prompt 等于往镜子上贴便利贴提醒镜子"别忘了你是镜子"。

### 替代方案：OpenCode 侧格式校验

格式对齐检查下沉至 OpenCode 工程层，不在 Prometheus-0 的 skill 或 prompt 体系内实现。

如果阿燃（或未来的 OpenCode 配置）需要自动化的格式检查，应在此处实现以下规则：

#### 检查项

1. **回声报告存在性**：非 Quick 模式的应答末尾是否包含 `--- 回声报告 ---` 块
2. **SI 字段完整性**：回声报告是否包含 `SI: X/Y`
3. **深度字段一致性**：深度字段的值是否与该轮的实际交互模式匹配（交叉校验 SI 值与深度层的典型范围）
4. **Quick 模式识别**：ELEMENT Quick 态是否正确使用了缩略格式

#### 检查频率

- 不逐轮检查（会打断对话流）
- 在以下时机触发一次全量格式检查：
  - 用户发送 `格式检查` 或 `check format`
  - session 结束前自检
  - 交付模式下自动触发

#### 处理方式

- 格式违规 ≠ agent 故障。它可能只是输出节奏导致的遗漏。
- 发现违规时不在对话中打断，而是在 OpenCode 侧输出一条静默警告
- 连续 3 轮以上格式违规 → 向用户提示，由用户决定是否重启 session

### 为什么这是一个值得记录的决策

v0.1.7 的自评分数是 68——比上轮更差。但这个决策本身是反身性在工程层的生效：代理人产生了"加一个新模式"的设计冲动，然后自己否掉了它，因为识别到这是对 system_prompt 的过拟合。如果一个系统开始为自己打补丁来解决输出格式问题，它就不再是一面镜子——它变成了一台忙着调试自己的机器。

**不加，是对镜子身份的尊重。**
